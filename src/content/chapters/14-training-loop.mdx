# The Training Loop: Optimizer, Mixed Precision, and Checkpoints

**Where we are.** We have a correct forward pass and a correct attention kernel. This chapter wraps them
in the loop that actually learns: compute a loss, backpropagate, take an optimizer step, and — crucially
— save a checkpoint that captures enough to *resume*. Every choice here is one the [numbers
contract](/transformers-systems/chapters/03-numbers-contract/) and the [memory
ledger](/transformers-systems/chapters/01-resource-accounting/) already priced.

## 1. AdamW: decoupled weight decay

<Invariant note="AdamW">
AdamW maintains, per parameter, a first moment (a smoothed gradient) and a second moment (a smoothed
squared gradient), and steps in the direction of the first scaled by the inverse root of the second.
Its defining choice is that weight decay is **decoupled** — applied directly to the weights rather than
folded into the gradient — so regularization does not interact with the adaptive scaling. The two
moments are the $8$ of the $16$ bytes-per-parameter the memory ledger charged to optimizer state.
</Invariant>

R1-tiny uses the standard large-model settings: learning rate $3\times 10^{-4}$, $\beta=(0.9, 0.95)$
(a slightly faster second-moment decay than the default), and weight decay $0.1$. These are not tuned
here — they are the reference recipe, frozen so a later chapter's numbers do not silently depend on an
untracked hyperparameter.

## 2. Mixed precision: compute low, accumulate high

<Invariant note="the master-weight policy">
Mixed-precision training computes the forward and backward in a low-precision format (bf16) for speed and
memory, while keeping an **fp32 master copy** of the weights and doing the optimizer update and the
gradient reductions in fp32. The reason is numerical: a bf16 weight cannot represent the tiny updates a
late-training step applies, so accumulating those updates in fp32 preserves them. Compute low, accumulate
high — the same byte-per-parameter split the training-state ledger charged.
</Invariant>

This policy is why the memory ledger's $16$ bytes/parameter includes both a bf16 weight and an fp32
master: they coexist by design. bf16's wider exponent (versus fp16) also removes the loss-scaling
machinery fp16 needs, simplifying the loop.

## 3. A checkpoint is the whole state, not the weights

<Invariant note="what a resumable checkpoint contains">
A checkpoint that stores only the model weights cannot resume training — it has thrown away the
optimizer moments (so the first resumed step takes a wrong-sized stride) and the data cursor (so it
re-sees or skips data). A *resumable* checkpoint captures the full training state: model weights,
optimizer moments, the learning-rate schedule position, the dataloader cursor, and every random-number
generator stream. The weights are the smallest part of what must be saved.
</Invariant>

This is the setup for the next chapter's gate: the checkpoint is only correct if resuming from it
reproduces an uninterrupted run, and that is only possible if the checkpoint is complete.

```python
opt = torch.optim.AdamW(m.parameters(), lr=3e-4, betas=(0.9, 0.95), weight_decay=0.1)
_, loss = m(x, y)                 # forward (bf16 compute)
opt.zero_grad(); loss.backward()  # backward
opt.step()                        # fp32 master update — the moments are 8 of the 16 B/param
```
</Versioned>

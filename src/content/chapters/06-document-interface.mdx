# The Document-AI Interface: One Tokenizer, Two Modalities

**Where we are.** The [tokenizer](/transformers-systems/chapters/05-tokenization/) reserved eight
document-structure tokens it never used on text. This chapter is why they exist. The document-AI model
R3 — the guide's north star — reads a page image and *writes structured markup*, and the decisive
design choice is that it writes it with the **same tokenizer and the same decoder** as the text model.
This chapter defines that interface; the systems that build and serve it are Part 7.

## 1. Structured output is a token sequence

<Invariant note="layout as a sequence-modeling problem">
A document's structure — its regions, tables, formulas, reading order — is a *tree*, but a decoder
emits a *sequence*. The interface that unifies them is to serialize the tree into tokens: structural
tags open and close nodes (`<table><tr><td>…</td></tr></table>`) — whether a given tag is one atomic
vocabulary entry or spelled out as ordinary text is a per-vocabulary decision, not part of the
framing — and geometry, where needed, is
emitted as **coordinate tokens** — real-valued positions quantized into vocabulary entries. Under this
framing, "understand this page" and "predict the next token" are the *same* operation, so document
understanding needs no separate detection head — only the reserved vocabulary and the decoder we
already have.
</Invariant>

This is the lineage the public literature established: **Pix2Seq** cast object detection as
next-token prediction over quantized coordinates; **Donut** and **Pix2Struct** made document parsing
OCR-free by predicting structured strings directly from pixels; **Nougat** predicted a full markup
document (headings, tables, math) from an academic page. R3 stands on this analogue — layout emitted
as tokens, never as a bespoke non-sequence output — which is precisely why one decoder suffices.

## 2. The document-control vocabulary is the seam

<Invariant note="a shared namespace binds the two models">
Because the structural tags live in the *same* tokenizer as the text tokens, the boundary between the
text model and the document model is a **vocabulary partition**, not a model boundary. The text model
R1 exercises the language tokens; the document model R3 exercises those *plus* the reserved
`<page> </page> <region> <table> <tr> <td> <formula> <eos_doc>` tags. Sharing the namespace means R3 inherits R1's
learned text distribution for free and adds only the ability to place structure around it.
</Invariant>

Concretely, a document target is a token stream that interleaves content and structure — a `<page>`
opens and `</page>` closes, `<region>` and `<table>` introduce their content, `<formula>` marks math,
`<eos_doc>` terminates. The tree is reconstructed downstream by matching opens to closes and by
containment — with one as-built nuance: only `</page>` is a *reserved* closing token, so any other
closer in the stream is ordinary byte-BPE text the decoder must spell out (ADR 0002). The
decoder's job is only to emit the sequence. This "layout ≠ text, but both are tokens" separation is
what lets the pilot overfit a single document target end-to-end before any of the Part-7 machinery exists:

<Renewable last_measured="2026-07-17" measured_on="Tier-0 RTX 2070 SUPER" source="https://github.com/brandon-behring/transformers-systems-lab">
The VLM interface spike drives a seeded random tensor (a stand-in for a rendered page) through a
small vision encoder and projector into the R1-tiny decoder and overfits one (image, markup) pair
until **fixed-length** greedy decoding reproduces the exact target sequence. What it validates is the
shared tokenizer and the structural-tag round-trip: the target mixes reserved control ids (`<page>`,
`<formula>`, `</page>`) with a text-encoded lookalike closer (`</formula>`), and the decoder emits
both with no architectural change beyond the input projection. `<eos_doc>` termination logic does not
exist in the spike — generation runs a fixed number of steps — per the pilot deviations register
(ADR 0002).
</Renewable>

## 3. What "done" means: tasks and benchmarks

An interface is only as real as the tasks it is scored on. The public document-AI benchmarks fix the
targets: **FUNSD** and **CORD** for form and receipt structure, **DocVQA** and **DUDE** for
document question answering, and long-context suites for multi-page reasoning. They matter here for a
methodological reason from Part 0: they define the *denominators* and the *quality target* against
which R3's time-to-quality is measured, so "the document model works" becomes a number, not an
impression.

<Versioned as_of="2026-07-17" version="torch==2.13.0">

```python
# A document target is just tokens from the shared vocabulary. Every tag below is
# an atomic reserved id; </page> is the only reserved closer, so inner nodes end by
# containment. The cell reads "42", not "$42" — a lone $ would violate the grammar's
# balanced $…$ math-delimiter rule.
target = "<page><region>Invoice<table><tr><td>Total<td>42</page><eos_doc>"
ids = tok.encode(target).ids     # same tokenizer as the text model — no re-tokenization
```
</Versioned>

**Where this goes.** The *text side* of R0 is now in place: a licensed corpus, a rebuildable
tokenizer, and a reserved document-control vocabulary shared across both models. The document side
remains design-forward per the pilot deviations register (ADR 0002): the train/val splits and
contamination probe, the document corpus itself, the §2.4 render contract, the §2.5 output grammar,
and the §2.6 image-token budget. The last Part-1 chapter turns tokens into the fixed-shape batches
training consumes, closing the text data contract.

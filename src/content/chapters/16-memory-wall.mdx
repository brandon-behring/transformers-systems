# The Memory Wall and Activation Checkpointing

**Where we are.** Part 3 trained R1 on one device. Part 4 scales it out — and the reason we *must* is
almost always memory, not compute. This chapter names the wall and introduces the one technique that
moves it without buying hardware: trading compute to recompute activations instead of storing them.

## 1. The memory wall

<Invariant note="three claims on device memory, in fixed proportions">
Training memory has three tenants. **Model state** — weights, gradients, optimizer moments — is fixed by
the parameter count (Part 0's $16$ bytes/parameter) and independent of batch. **Activations** — the
intermediate tensors the backward pass needs — scale with $\text{batch}\times\text{sequence}\times
\text{layers}\times d_\text{model}$ and are the *variable* tenant. **Fragmentation and workspace** take
the rest. At scale, activations dominate and grow with the very knobs (batch, context) you want to
increase — so memory, not FLOP/s, is the first wall.
</Invariant>

The wall is why R1-tiny is *tiny*: its fixed model state is 101 MB, but its activations at the training
batch are several times that, and the sum must fit 8 GB. Growing the model or the context first exhausts
memory, not compute — the opposite of the intuition that "a bigger GPU is a faster GPU."

## 2. Activation checkpointing: recompute instead of store

<Invariant note="the compute-for-memory trade">
Activation (gradient) checkpointing stores only a subset of activations on the forward pass — typically
one tensor per layer boundary — and *recomputes* the rest during the backward pass from those saved
points. It trades a second forward pass (extra compute) for a large cut in activation memory. In the
common per-layer scheme — one checkpoint per layer boundary — stored activation memory stays
proportional to the layer count $L$ but with a large constant reduction, since only the boundary tensors
and one layer's recompute workspace are live. The $\Theta(\sqrt{L})$ memory result (Chen et al.) comes
from a different schedule: a checkpoint every $\sqrt{L}$ layers, recomputing one $\sqrt{L}$-layer segment
at a time during the backward. Either way the price is roughly one extra forward. The knob is *which*
activations to keep; selective schemes recompute only the cheap-to-recompute, memory-heavy ones.
</Invariant>

This is a direct application of the roofline's logic: if a workload is memory-bound and compute-idle,
spending idle compute to buy memory is free real estate. Korthikanti's selective recomputation is the
refined form — cutting most activation memory while adding well under a full extra forward.

## 3. When recompute runs out: shard

Activation checkpointing buys headroom on *one* device; it does not let a model whose *weights* exceed
one device fit. When the model state itself is too large — or when even checkpointed activations plus
state overflow — the only remaining move is to split the model across devices. That is the parallelism
ladder of the next chapter, and activation checkpointing composes with every rung of it.

<Renewable last_measured="2026-07-17" measured_on="Tier-0 RTX 2070 SUPER" source="https://github.com/brandon-behring/transformers-systems-lab">
R1-tiny's measured run makes the proportions explicit: the fixed model state is the run's 101.2 MB train
state, while measured peak memory at the training batch (bs=8, fp32) is 1,171,365,888 B
$\approx 1.17$ GB — roughly 1.07 GB of activations and workspace on top of the fixed state. Activations
are the larger, batch-driven tenant, and the sum is what the 8 GB budget must hold. This is the memory
wall in miniature: the pilot is sized so state + activations fit, and any growth hits memory first.
</Renewable>

**Where this goes.** When one device is not enough, the model is split across many. The next chapter is
the taxonomy of *how* — the five axes of parallelism and what each one costs in communication.

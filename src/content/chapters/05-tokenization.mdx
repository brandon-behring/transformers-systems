# Tokenization: A Byte-Level BPE You Can Rebuild

**Where we are.** The [corpus](/transformers-systems/chapters/04-corpus/) is text; the model consumes
integers. The tokenizer is the map between them, and in R0 it is a *rebuildable artifact* — trained by
a script, checksummed into the manifest, and shared between the text model and the document-AI model.
This chapter builds it from the byte up.

## 1. BPE over bytes

<Invariant note="byte-level BPE, stated exactly">
Byte-Pair Encoding starts from an alphabet and greedily merges the most frequent adjacent pair into a
new symbol, repeating until the vocabulary reaches a target size. Taking the alphabet to be the
**256 possible bytes** makes the tokenizer *total*: every possible input encodes without an
out-of-vocabulary token, because every string is a sequence of bytes. There is no UNK, no
normalization surprise, and no language the tokenizer cannot represent — only more or fewer merges
spent on it.
</Invariant>

The consequence is robustness: emoji, code, non-Latin scripts, and corrupted bytes all round-trip.
The cost is that rare content is spent one byte at a time, which is exactly the trade the *vocabulary
budget* manages.

## 2. The vocabulary is a budget

A vocabulary size is not a free parameter to maximize; it is a budget to allocate. R0 fixes it at
**8,192**, and that number is spent in three places: the 256 base bytes (unconditional), the reserved
special tokens (below), and the remaining merges that buy shorter sequences for common text. A larger
vocabulary shortens sequences (fewer tokens per document, so cheaper attention) but enlarges the
embedding table — recall from the [parameter ledger](/transformers-systems/chapters/01-resource-accounting/)
that R0's tied embedding is $V d_\text{model}$, one third of R1-tiny at $V=8192$. Doubling the vocab
would double that third. The budget is a direct dial between sequence length and parameter count.

## 3. Special tokens reserve the future

<Invariant note="special tokens are a namespace decision">
Special tokens are vocabulary entries the trainer will never split or merge away — control symbols
with fixed ids. Reserving them at tokenizer-training time buys two things: **merge-avoidance** (each
tag string encodes atomically as one id, and no learned merge can cross or absorb it) and a **fixed
embedding size** (the table never has to grow when the document model arrives). What reservation
does *not* buy is meaning: a control token the text model never sees during training carries no
learned embedding either way — that is a training-data question, not a vocabulary question.
</Invariant>

R0 reserves ten, at ids 0–9: `<|endoftext|>` and `<|pad|>` for sequences, and
`<page> </page> <region> <table> <tr> <td> <formula> <eos_doc>` for document structure. Note the
asymmetry: only `</page>` gets a reserved closing tag, and the contract's `$…$` math-delimiter tokens
are not reserved — other closers and the delimiters encode as ordinary byte-BPE text. Reserving up
front is itself a pilot deviation: the frozen contract's production path *appends* the
document-control vocabulary to R1's finished tokenizer with mean-initialized embeddings, per the
pilot deviations register (ADR 0002). The text model R1 will only ever emit the first two — but
because the document tokens share the *same* tokenizer and ids, the document model R3 needs no
re-tokenization and no vocabulary resize. One tokenizer, two
modalities; the seam is the subject of the [next chapter](/transformers-systems/chapters/06-document-interface/).

<Renewable last_measured="2026-07-17" measured_on="Tier-0 RTX 2070 SUPER" source="https://github.com/brandon-behring/transformers-systems-lab">
R0 trains a byte-level BPE tokenizer to **vocab 8,192** on the checksummed corpus with the ten
special tokens reserved at ids 0–9, and records its `tokenizer_sha256` in the manifest. Re-running
the build on the same corpus bytes with the same `tokenizers` build reproduces the same hash —
deterministic given its inputs, though the library version itself is unpinned (ADR 0002).
</Renewable>

<Versioned as_of="2026-07-17" version="torch==2.13.0">
```python
from tokenizers import Tokenizer, models, trainers, pre_tokenizers
tok = Tokenizer(models.BPE(unk_token=None))          # no UNK — byte alphabet is total
tok.pre_tokenizer = pre_tokenizers.ByteLevel(add_prefix_space=False)
tok.train([CORPUS], trainers.BpeTrainer(
    vocab_size=8192,
    special_tokens=["<|endoftext|>", "<|pad|>", "<page>", "</page>", "<region>",
                    "<table>", "<tr>", "<td>", "<formula>", "<eos_doc>"],
    initial_alphabet=pre_tokenizers.ByteLevel.alphabet()))   # all 256 bytes
```
</Versioned>

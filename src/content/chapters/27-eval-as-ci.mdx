# Eval-as-CI and Statistical Rigor

**Where we are.** [Checkpoint selection](/transformers-systems/chapters/25-peft-valloss/) needs a metric,
and [RLVR](/transformers-systems/chapters/26-preference-optimization/) needs a verifier. Both are the same
thing: an automated evaluation that runs every change. This chapter makes that evaluation a CI citizen and
gives it the statistical rigor without which "the model improved" is a guess.

## 1. Evaluation is continuous integration

<Invariant note="eval-as-CI">
An evaluation that is run occasionally, by hand, on an ad-hoc set is not a control — it is an anecdote. An
evaluation is a *control* when it runs automatically on every change, against a **frozen, versioned**
dataset, and blocks a regression the way a failing unit test blocks a merge. This is the numbers contract
of Part 0 applied to behavior: the eval set is checksummed, the metric is deterministic, and a drop is a CI
failure, not a discussion.
</Invariant>

The document model is designed to make this concrete: its structural metric — does the produced markup
match the ground-truth structure — is specified to run on a frozen document set every checkpoint, with any
checkpoint that regresses it rejected automatically. Selection, RLVR reward, and CI gate are one metric by
design.

## 2. Contamination invalidates the measurement

<Invariant note="the eval set must be disjoint from training">
A benchmark score is meaningless if the benchmark leaked into training: the model then *recalls* the answer
instead of *solving* the task, and the number measures memorization, not capability. Guarding against it is
a data-pipeline discipline, and it is why the frozen contract (`reference-model.md` §2.1) specifies
train/val splits plus a held-out contamination probe — split by document id, frozen, hashed, with val and
probe ids excluded from training and external eval sets n-gram-decontaminated against the corpus. The
eval's validity is a *property of the data contract*, established before training, not a hope checked
after. The as-built pilot R0 has none of these splits yet (ADR 0002, deviation 3); building them is the
prerequisite for any trustworthy eval number.
</Invariant>

## 3. Statistical rigor: is the gain real?

<Invariant note="report distributions and test significance">
A single eval number is a sample. Two checkpoints whose scores differ by less than the measurement's noise
are not distinguishable, and shipping the "better" one is a coin flip dressed as progress. Rigor means
reporting a metric with a confidence interval (e.g. bootstrapped over the eval set), running enough seeds to
estimate run-to-run variance, and applying a significance test before claiming an improvement — with a
correction when many variants are compared, or the best-of-many will look good by chance alone. This is
Part 0's "report a distribution, not a point," now guarding *quality* rather than *throughput*.
</Invariant>

<Renewable last_measured="2026-07-17" measured_on="Tier-0 RTX 2070 SUPER" source="https://github.com/brandon-behring/transformers-systems-lab">
The guide *practices* eval-as-CI on itself, with an honest split: hosted CI runs the numbers-contract
validator, the golden accounting fixture, and the R0 batch golden test, failing the build on any drift;
the kernel-parity and resume-gate checks run locally on Tier-0, and the kernel test self-skips loudly on
the GPU-less hosted lane rather than pretending to pass — the same discipline this chapter prescribes for
model quality, applied to the guide's own numbers. The model-quality eval harness (structural metric on a
frozen document set) is specified as the R3 selection-and-reward signal; it is design-forward on Tier-0,
reusing this machine.
</Renewable>

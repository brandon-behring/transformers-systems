# PEFT and the Val-Loss Trap

**Where we are.** [SFT](/transformers-systems/chapters/24-sft/) adapts a model by updating its weights —
expensive to store and slow to iterate. This chapter makes adaptation cheap with a low-rank overlay, and
then delivers the lesson cheap iteration exposes: the loss you train is a *surrogate*, and selecting on it
can ship the wrong model.

## 1. LoRA: train the overlay, freeze the backbone

<Invariant note="low-rank adaptation">
LoRA freezes the pretrained weight $W$ and learns a low-rank *overlay* $BA$ added on top, so the effective
weight is $W + BA$ with only $B$ (tall-skinny) and $A$ (short-wide) trainable. For a $d\times d$ projection
at rank $r$, the trainable count drops from $d^2$ to $2rd$ — at $r \ll d$ a reduction of hundreds of times.
Because the frozen backbone is shared, one can keep a *different* small overlay per task and swap them
without touching the expensive base weights.
</Invariant>

The engineering payoff is iteration speed: a LoRA fine-tune trains a tiny fraction of the parameters, fits
far more easily in memory (no optimizer state for the frozen weights), and produces a small artifact. This
is what makes running *many* fine-tunes — and therefore the selection problem below — practical.

The arithmetic is exact and checkable on R1-tiny. A rank-$8$ overlay on one $256\times256$ attention
projection trains $2\cdot 8\cdot 256 = 4{,}096$ parameters against $65{,}536$ for the full matrix — a
$16\times$ reduction. On the tied $8192\times256$ vocabulary matrix the same rank trains
$8\cdot(8192+256) = 67{,}584$ against $2{,}097{,}152$ — a $31\times$ reduction, essentially at the
$256/8 = 32$ ceiling the smaller dimension imposes on any rank-$8$ overlay. The wide matrix is where the
most *absolute* trainable parameters are saved (about $2.03$ M fewer than full fine-tuning of it), but its
reduction *ratio* is only about twice the square projection's — capped, not unbounded, by the narrow
dimension.

## 2. The val-loss trap

<Invariant note="the loss is a surrogate for the metric">
The quantity you optimize (validation cross-entropy) is not the quantity you ship (task accuracy — for the
document model, correct *structure*). The two decouple for concrete reasons: fine-tuning overfits to a
narrow demonstration distribution, so held-out per-token loss stops tracking behavior outside it; and the
loss is measured *teacher-forced* (every prediction conditioned on a ground-truth prefix) while the shipped
model runs *free-running* on its own output. The decoupling is documented in both directions — LIMA reports
validation perplexity *rising* while generation quality improves — so validation loss is not even a
reliably-signed proxy. **Selecting the checkpoint with the lowest validation loss can therefore ship a
worse model than one selected on the metric itself.**
</Invariant>

The fix is a discipline, not a hyperparameter: run the *real* metric inside the training loop and select
the checkpoint that scores best on **it**, not on the loss. This is the single most common way a
fine-tuning pipeline silently underperforms — it optimizes and selects on the surrogate throughout, and
never looks at the metric until it is too late.

## 3. Why this belongs to systems, not just modeling

Selecting on the metric means the metric must *run in CI* — cheaply, deterministically, every checkpoint —
which is a systems requirement: the evaluation has to be fast enough to run inline and reproducible enough
to compare across runs. The val-loss trap is thus the bridge to the eval-as-CI chapter: checkpoint
selection is only as trustworthy as the automated metric behind it.

<Renewable last_measured="2026-07-17" measured_on="Tier-0 RTX 2070 SUPER" source="https://github.com/brandon-behring/transformers-systems-lab">
LoRA fine-tuning is **design-forward** on Tier-0 — the pilot's measured run is full-parameter pretraining,
and no overlay has been trained. The val-loss-trap discipline (select on the metric) is likewise specified
as a checkpoint-selection policy, not yet an exercised loop; the metric it would select on is the subject
of the eval-as-CI chapter.
</Renewable>

**Where this goes.** SFT and PEFT teach *format* from demonstrations. But some behaviors are easier to
*judge* than to *demonstrate* — and for those we optimize against a preference or a verifier, the subject
of the next chapter.

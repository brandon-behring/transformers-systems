# The Data Pipeline: Packing, Batching, and Immutability

**Where we are.** We have a licensed corpus and a tokenizer that reserves document structure. The last
step of the R0 contract is the one training actually reads: **fixed-shape batches of token ids**. This
chapter turns a ragged stream of documents into a rectangular array, and — as with every R0 artifact —
checksums it so the training input is immutable.

## 1. Packing: from ragged documents to a rectangle

<Invariant note="sequence packing, stated exactly">
Documents have unequal lengths; a training batch must be a rectangle of shape (sequences × context).
**Packing** bridges the two: concatenate encoded documents into one long token stream, insert a
separator (`<|endoftext|>`) at each document boundary, then slice the stream into fixed-length rows.
Packing wastes no tokens on padding — every position carries signal — at the cost of one subtlety: a
row can span a document boundary, so the separator token is what tells the model where one document
ends and the next begins.
</Invariant>

Padding-based batching, the alternative, pads every sequence to the longest in the batch and wastes
compute proportional to the length variance. For a corpus of short, varied documents that waste is
large, which is why packing is standard for pretraining. The concat-and-slice convention R0 uses is
the GPT-2/GPT-3 lineage — documents joined by an end-of-text delimiter and cut into full context
windows (Brown et al., 2020). A stricter variant instead bin-packs whole sequences into rows and
separates them with attention masks (Kosec et al., 2021), avoiding cross-document attention at the
cost of pipeline complexity. The separator is not cosmetic: without it the
model would learn spurious continuations across unrelated documents.

## 2. The dtype is a function of the vocabulary

A token id is an integer, and the integer width is a decision with a memory cost. R0's vocabulary is
**8,192**, which fits in 16 bits, so the batch array is stored as `uint16` — half the footprint of the
`int32` a naïve pipeline would use. The rule is exact: `uint16` holds ids $0$ through $65{,}535$, so
any vocabulary of size at most $2^{16}=65{,}536$ fits; cross that threshold and the dtype must
widen. Choosing the narrowest dtype that holds the
vocabulary is the same byte-discipline the [memory ledger](/transformers-systems/chapters/01-resource-accounting/)
applied to weights, now applied to data.

## 3. The whole pipeline is one immutable artifact

<Invariant note="verification is transitive through checksums">
The batches are a deterministic function of the corpus and the tokenizer, each of which is itself
checksummed. Recording all three hashes — corpus, tokenizer, batches — in one manifest makes
*verification* transitive: anyone who re-runs the pipeline can check, bit for bit, whether they
obtained the identical training input. Detection is not re-derivation: the pilot pins no upstream
dataset revision and leaves its libraries unpinned (ADR 0002), so a divergent re-run is *caught*,
not prevented. A model's claims are only as reproducible as the
data artifact they were trained on, and here that artifact has a single immutable name.
</Invariant>

<Renewable last_measured="2026-07-17" measured_on="Tier-0 RTX 2070 SUPER" source="https://github.com/brandon-behring/transformers-systems-lab">
R0 packs a prefix slice of the tokenized corpus — the first 131,072 tokens (256 × 512), roughly
1–2% of the assembled bytes — into a **(256 × 512)** `uint16` array and records `batches_sha256`
alongside the corpus and tokenizer hashes in `r0/manifest.yml`. The golden-batch test
(`tests/test_r0_batches.py`) asserts the shape **(256, 512)**, the `uint16` dtype, that every token
id is below the vocabulary size 8,192, and that the file's SHA-256 matches the manifest entry — so a
pipeline change that alters the training input trips lab CI rather than silently retraining on
different data.
</Renewable>

vocabulary, and these batches — is now a checksummed contract. The rest stays design-forward per the
pilot deviations register (ADR 0002): the §2.1 train/val splits and contamination probe, the
